# 后端 API 与服务改动规则（MVP）

## 路由与组织

- 新接口优先放入 `src/routes/` 下已有模块；只有在职责明显独立时才新增路由文件。
- 路由注册统一在 `src/index.ts` 中完成，避免分散挂载。
- 已有接口路径风格优先保持一致，例如：
  - `/api/reports/*`
  - `/api/tasks/*`
  - `/files/*`

## 返回结构

- 对任务类接口，优先保持现有返回结构语义一致：`taskId`、`status`、`content`、`detail`。
- 非必要不要随意改字段名；如果必须新增字段，优先追加而不是替换。
- 错误返回继续使用 `c.json({ error, ... }, statusCode)` 的直观风格。

## 数据校验与文档

- 涉及模板数据时，优先通过 `templateManager`/`zod` 做校验，不把校验逻辑散落在多处。
- 如果 API 行为发生变化，记得同步检查 `src/openapi/spec.ts` 是否需要更新。
- 不要为了“看起来更 REST”而在 MVP 阶段大改现有路径命名。

## 改动边界

- 优先最小化修改已有路由行为，避免影响 dashboard 与现有调用方。
- 变更前先确认调用入口、返回结构和状态码，再下手修改。
