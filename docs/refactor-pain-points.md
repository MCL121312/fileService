## 重构痛点整理

本文档用于记录当前项目里仍然遗留的旧痛点，方便后续继续重构时参考。

说明：这份内容是结合当前代码结构与你补充的想法反推出来的，不是严格的历史设计文档。

## 当时重构的核心目标

根据现有代码和你的补充，当时更真实的目标大致是这 3 条：

1. 只支持 PDF，不继续扩展 Word
2. 简化模板目录和模板接入方式
3. 改造浏览器管理，让浏览器池更正确、更稳定

## 当前状态概览

从当前 `main` 看，三个目标的完成度大致如下：

- 只支持 PDF：基本已经达成
- 模板目录更简洁：只做了一部分，旧痛点还在
- 正确的浏览器池：有改进，但仍然是主要遗留问题

## 仍然遗留的旧痛点

### 1. 浏览器池还不够“正确”

这是当前最值得继续处理的一项。

当前 `src/core/browserPool.ts` 已经不是简单 demo，但离“正确的浏览器池”还有差距：

- 仍然采用 `create -> init -> use` 的两阶段初始化模型
- 调用方依赖启动阶段先执行 `browserPool.init()`
- 池中实例借出前没有健康校验
- 没有明确的坏实例淘汰策略
- 模块同时承担了浏览器发现、可用性检测、池管理多种职责

这会带来几个实际问题：

- 生命周期依赖顺序，启动链路容易变脆
- 如果浏览器实例异常关闭，后续可能仍被复用
- 问题往往在生成 PDF 时才暴露，而不是借出实例时暴露
- 池有了，但稳定性还不算真正可控

### 2. PDF 渲染链路仍然偏脆弱

`src/core/pdfRenderer.ts` 当前的思路是：

- 读取 HTML 模板
- 注入 Vue runtime 和数据
- `page.setContent()`
- `waitForSelector('#app')`
- 额外等待 100ms
- 导出 PDF

这套链路能工作，但仍有明显经验值成分：

- 渲染完成时机判断不够强
- `waitForSelector('#app') + 固定延时` 偏脆弱
- 数据复杂或页面渲染慢时，可能出现偶发不稳定
- Vue runtime 通过字符串注入，后续调试成本较高

如果浏览器池是“基础设施稳定性”问题，那么这里就是“渲染完成时机稳定性”问题。

### 3. 模板系统仍然依赖扫描目录和命名约定

当前模板接入方式仍然主要依赖 `src/core/templateLoader.ts`：

- 启动时扫描 `src/templates/`
- 查找 `*.schema.ts`
- 再匹配对应的 `*.html`

这意味着模板仍然是“被发现”的，而不是“被显式注册”的。

遗留问题主要有：

- 从入口看不容易直接知道系统加载了哪些模板
- 模板接入依赖约定，不够直观
- 出问题时需要顺着 loader 逻辑追查
- 模板命名和文件组织仍然受到 loader 协议约束

这部分虽然功能上能跑，但从重构目标看，还没真正达到“更简洁”的状态。

### 4. 模板目录看起来不复杂，但职责边界还不够干净

当前模板结构仍然偏向：

- `pdf.schema.ts`
- `pdf.html`
- 由 loader 在运行时拼装成模板能力

问题不在于文件数量多，而在于职责仍然有些拧巴：

- 模板本身不是独立注册单元
- 模板层仍然受 core loader 类型影响
- 模板结构更像“文件协议”，不是“清晰语义结构”
- 公共资源与模板资源边界也不算特别收敛

比如 `src/templates/health-report/pdf.schema.ts` 仍然依赖 `src/core/templateLoader.ts` 中定义的类型，这说明模板层和核心加载层还没有完全解耦。

### 5. 虽然已经只支持 PDF，但内部还残留少量“格式抽象壳”

从接口行为上，当前系统已经回到单格式 PDF：

- `src/types/task.ts` 中 `OutputFormat` 只有 `pdf`
- `src/routes/reports.ts` 明确拒绝非 PDF 请求
- `src/routes/files.ts` 只接受 `.pdf`
- `src/openapi/spec.ts` 里也只暴露 PDF

但内部任务流里仍保留了一点“格式抽象”的形状，例如：

- `task.format`
- `FORMAT_EXTENSIONS`
- `FORMAT_CONTENT_TYPES`
- `reportGenerator.generate(templateId, data, format)`

这些现在不算严重问题，但说明系统虽然已经收缩回 PDF-only，内部还有一点未完全瘦身的痕迹。

### 6. 文档、模板现状与实际实现之间还有轻微错位

当前 OpenAPI 示例里仍能看到多个模板示例，例如 `guide-sheet`，但当前模板目录下实际可见内容并不完全对应。

这类问题不会立刻导致服务报错，但会带来：

- 文档理解成本升高
- 维护者对当前真实能力的判断不够直接
- 继续重构时容易基于旧认知继续改动

## 按优先级排序

### 高优先级

1. 浏览器池缺少健康校验和坏实例淘汰机制
2. PDF 渲染完成时机判断偏脆弱

### 中优先级

3. 模板系统仍依赖扫描目录与命名约定
4. 模板层与 core loader 之间还有反向耦合

### 低优先级

5. 内部任务流还残留少量格式抽象壳
6. 文档和模板现状存在轻微错位

## 如果继续重构，建议的顺序

建议优先顺序：

1. 先处理浏览器池正确性
2. 再提升 PDF 渲染链路稳定性
3. 然后整理模板目录和模板接入方式
4. 最后再做内部抽象瘦身和文档对齐

原因很简单：

- 浏览器池和 PDF 渲染稳定性，直接影响系统是否可靠
- 模板目录问题主要影响可维护性和后续扩展体验
- 内部抽象瘦身和文档对齐属于后收口工作

## 一句话总结

当前项目已经基本回到“只支持 PDF”的正确方向，但模板接入方式还不够简洁，浏览器池也还没有完全达到“正确、稳定、可控”的状态。
