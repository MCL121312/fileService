# FileService Agent Guide

## 项目概况

- 本项目是一个基于 `Hono + TypeScript + SQLite` 的报告/文件生成服务。
- 服务入口是 `app.ts`，应用装配入口是 `src/index.ts`。
- 路由位于 `src/routes/`，核心能力位于 `src/core/`，模板入口位于
  `src/templates/index.ts`。
- dashboard 前端资源位于 `public/`。

## 通用工作方式

- 始终使用中文沟通。
- 优先做渐进式修改，不做无关重构。
- 优先复用现有目录、函数和数据流，不平行再造一套结构。
- 如果用户只要求 MVP，先交付最小闭环版本，再单独列出可选增强项。
- 如果需求不完整，先确认影响范围，再进行结构性修改。

## 后端改动规则

- 新接口优先放入现有 `src/routes/` 模块中，避免随意新增并行入口。
- 路由统一在 `src/index.ts` 注册。
- 任务类接口尽量保持现有返回结构语义稳定：`taskId`、`status`、`content`、`detail`。
- 非必要不要修改已有字段名；如果必须扩展，优先追加字段而不是替换字段。
- 如果 API 行为发生变化，同步检查 `src/openapi/spec.ts`。

## 模板与任务流规则

- 模板相关能力优先通过 `templateManager` 接入。
- 任务创建、排队、结果读取、清理优先复用 `taskManager`。
- 不要在 route 中直接塞入重型生成逻辑，route 应主要负责参数接收与响应组装。
- 新功能优先沿用“模板 -> 任务 -> 文件”的现有链路。

## 前端与 CSS 规则

- dashboard 改动优先局部增强，不轻易大改 DOM 结构与交互节奏。
- 总是优先使用嵌套式 CSS。
- 在 CSS 中不要用 `&` 代替父级，除非是伪类或伪元素场景。
- 样式修改保持渐进式：先补局部 class，再补局部样式块，非必要不重排整份样式表。
- 前后端联动改动时，检查 `public/app.js` 与接口字段是否仍兼容。

## 验证要求

- 修改前先阅读相关文件，再下手编辑。
- 修改后优先做最小范围验证，避免高成本操作。
- 如果只改文档或规则文件，不需要跑无关测试。
- 如果改了 API、任务流或 dashboard，至少检查相关路径和字段是否一致。
